## 一、当前系统全貌

### 1.1 已实现的处理流水线

当前 `routers/chat.py` 实现了三阶段流水线，每次学生发消息按以下顺序处理：

```
学生消息
  │
  ▼
阶段 1: SessionManager 获取/创建会话 + 加载学生档案（JSON 持久化）
  │     + 快速情绪预筛（关键词检测 RED）
  │     + 自动判断教学模式（auto → diagnosis/explain/socratic）
  │
  ▼
阶段 2: LLM 状态评估（按需触发）
  │     前 3 条消息跳过，之后每 3 条触发一次
  │     StateEvaluator 调用 LLM → 输出 JSON:
  │       {cognitive_state, emotion, answer_quality, hint_level, reason}
  │     apply_evaluation() 更新 StudentProfile
  │
  ▼
阶段 3: RAG 检索 + 构建 system prompt + 流式 LLM 教学回复
        build_system_prompt() 按 mode 选模板 → append RAG 文本
        → provider.chat_stream() 逐 token 输出
        → 将回复加入会话 → 保存学生档案
```

### 1.2 各模块详细状态

#### `llm/prompts.py` — 三模板系统（旧版，需重构）

**当前内容**：
- `SOCRATIC_SYSTEM_PROMPT`（68 行）：苏格拉底式教学，含提问框架（聚焦性/可操作性/递进性）、追问策略（5 种学生回答类型）、例外规则（4 种可直接给答案的情况）、语气要求
- `DIAGNOSIS_SYSTEM_PROMPT`（38 行）：错误诊断，含 4 步诊断流程（确认理解→定位→分析→修复）、5 条关键规则
- `EXPLAIN_SYSTEM_PROMPT`（34 行）：概念讲解，含 5 步结构（直觉→为什么→例子→连接→检验）
- `build_system_prompt(mode, state_description, error_info, code_context, student_context)`：按 mode 选模板，用 `.format()` 注入变量

**问题**：
- 三套模板互相独立，无法根据学生具体状态动态调整引导策略
- RAG 检索结果在 `chat.py` 中直接 append 为纯文本，没有结构化注入
- 没有教学策略动态生成（如"学生情绪 YELLOW + 概念阶段 → 用类比"）
- 没有自我批评机制

#### `teaching/state_evaluator.py` — LLM 状态评估器（已完成）

**当前内容**：
- `EVALUATION_PROMPT`：让 LLM 分析学生回复，输出 JSON
- `StateEvaluator` 类：
  - `evaluate()` — 构建历史摘要（最近 5 条），调用 LLM（非流式，temperature=0.1，max_tokens=512）
  - `_extract_json()` — 4 种 JSON 提取策略（代码块→直接解析→正则匹配→宽松花括号）
  - `apply_evaluation()` — 将 JSON 结果映射到 StudentProfile 字段

**质量评估**：设计良好。JSON 提取有 4 层 fallback，解析失败时保持当前状态而非猜测。低温度保证输出稳定。

#### `teaching/cognitive_state.py` — 认知状态定义（已完成）

**当前内容**：
- `CognitiveState` 枚举：S1_CONCEPT / S2_CONVERTING / S3_CODING / S4_DEBUGGING
- `EmotionLevel` 枚举：GREEN / YELLOW / ORANGE / RED
- `StudentProfile` dataclass：
  - 字段：student_id, current_state, current_emotion, state_history, topics_seen, common_mistakes, hint_level, message_count
  - `update()` — 更新状态 + 追加历史（保留最近 20 条）
  - `get_state_description()` — 返回自然语言描述，注入 system prompt

**质量评估**：设计简洁，"只记录不判断"的哲学正确。`get_state_description()` 为每种状态/情绪/提示级别提供了具体的教学策略指引。

#### `teaching/error_diagnosis.py` — 错误诊断（已完成）

**当前内容**：
- `ParsedError` dataclass：error_type, error_message, file_path, line_number, traceback_text
- `ErrorDiagnosis` 类：

#### `teaching/emotion_detector.py` — 情绪预筛（已完成）

#### `state/student_profile.py` — 学生档案持久化（已完成）

**当前内容**：

#### `state/session_manager.py` — 会话管理（已完成）

#### `rag/indexer.py` + `rag/retriever.py` — 基础 RAG（已完成，需增强）

- 扫描 `knowledge-base/` 下的 `.md` 和 `.json` 文件

## 三、当前系统能做什么（能力边界）

### 3.1 能做的

1. **苏格拉底式对话**：学生提问 → AI 用提问引导，不直接给答案
2. **错误诊断**：学生粘贴 traceback → AI 按 4 步流程引导定位
3. **概念讲解**：学生要求解释 → AI 按 5 步结构讲解
4. **自动模式切换**：根据 context 自动判断 diagnosis/explain/socratic
5. **认知状态追踪**：LLM 推断学生处于 S1-S4 哪个阶段
6. **情绪检测**：关键词检测 RED + LLM 推断其他级别
7. **提示级别升级**：LLM 判断何时从 nudge 升级到 targeted_question / partial_reveal
8. **知识库检索**：向量相似度 + error_type boosting
9. **学生档案持久化**：JSON 文件存储，重启后恢复
10. **多 LLM 后端**：Claude / OpenAI / Ollama 可切换

### 3.2 不能做的（待实施）

1. **动态策略调整**：不能根据"学生情绪 YELLOW + 概念阶段 + 已见 input() 知识点"动态选择最佳引导方式
2. **结构化知识注入**：RAG 结果只是纯文本拼接，没有 reasoning_chain 引导推理
3. **自我纠错**：LLM 可能不小心给答案，系统不检查
4. **教学案例参考**：没有成功引导案例库供 LLM 参考
5. **概念知识索引**：没有结构化的概念知识（含常见误解和教学示例）
6. **VSCode 集成**：无法在 VSCode 中使用（扩展未完成）
